Inicjowanie (lokalnego) repozytorium

Inicjowanie (lokalnego) repozytorium (często w skrócie repo) oznacza katalog, który należy śledzić. Całość lub jej część można jawnie dodać do listy plików do śledzenia.

Komendy git wpisujemy w konsoli 'Git Bash', którą otrzymujemy po instalacji GIT-a na komputerze z adresu: git-scm.com 

$ cd project                      // przejdź do katalogu 'project'
$ git init                        // zainicjuj lokalne repozytorium
$ git add .                       // dodaj katalog, pojedyncze pliki do śledzenia
$ git commit -m "initial commit"  // zapisuje bieżącą lokalną migawkę (commit)

Następnie każda zmiana śledzonych plików, po zatwierdzeniu, zostanie zapisana jako nowa wersja lub migawka, jednoznacznie identyfikująca zmiany we wszystkich zmodyfikowanych plikach. Git jest wyjątkowo skuteczny i wydajny w archiwizacji pełnej historii projektu, między innymi poprzez przechowywanie tylko różnic między plikami.

Oprócz lokalnych kopii repozytorium łatwo jest utworzyć zdalne repozytorium na GitHub za pomocą interfejsu internetowego, a następnie zsynchronizować lokalne i zdalne repozytoria.

git push origin master          // przesyła lokalne zmiany do zdalnego repozytorium
git pull origin master          // pobiera zdalne zmiany do lokalnego repozytorium

Idąc za radą Tony'ego Rossiniego z 2005 r., Aby "wcześnie zatwierdzać, często popełniać i zatwierdzać w repozytorium, z którego możemy łatwo wycofywać swoje błędy", można organizować swoją pracę w małych skokowych zmianach. W każdej chwili można wrócić do poprzedniej wersji. W większych projektach wielu użytkowników może pracować na tym samym zdalnym repozytorium, a wszystkie składki są zapisywane, przywracane i przypisywane autorowi.

Ze względu na rozproszony projekt, każde aktualne lokalne repozytorium Git jest dokładną historyczną kopią wszystkiego, co zostało zatwierdzone - zmiany w plikach, dzienniki zdarzeń zatwierdzania itp. Te kopie działają również jako niezależne kopie zapasowe, obecne w pamięci każdego użytkownika urządzenia. Git można uznać za odporny na awarie z tego powodu, że jest scentralizowanym systemem kontroli wersji. Jeśli zdalny serwer GitHub jest niedostępny, współpraca i praca mogą być kontynuowane między użytkownikami, w przeciwieństwie do scentralizowanych alternatyw.

Interfejs sieciowy oferowany przez GitHub zapewnia przyjazne narzędzia do wykonywania wielu podstawowych operacji oraz łagodne wprowadzenie do bogatszego i bardziej złożonego zestawu funkcji.

Dostępność publicznego repozytorium zawierającego kod źródłowy nie powoduje, że oprogramowanie jest jako open-source. Powinieneś użyć licencji zatwierdzonej przez Open Source Initiative (OSI), która określa, jak oprogramowanie może być swobodnie używane, modyfikowane i udostępniane. Typowe licencje, takie jak te wymienione na stronie http://choosealicense.com są preferowane. Należy zauważyć, że plik LICENCJI w repozytorium powinien być plikiem tekstowym zawierającym treść licencji zatwierdzonej przez OSI, a nie tylko odniesienie do licencji.
